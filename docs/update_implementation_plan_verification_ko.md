- ❌ **INCORRECT** — 계획서 주장이 사실과 다름

## 종합 결론 (요약)

계획서는 **현재 코드의 실제 결함을 매우 정확하게 진단**하고 있습니다. 검증한 핵심 주장 약 30건 중 **약 28건이 사실 그대로**(✅), 1건이 부분 정정(⚠), 1건이 의문(❓)으로 분류됩니다. 구체적 파일 인용·라인 번호가 정확하고, 진단된 버그(예: `insert_images.py`의 `../images/` 하드코딩, 한국어 PDF의 "Chapter N" 레이블, 단일 언어 viewer의 secondary fallback)는 코드를 직접 보면 **모두 재현 가능한 실제 결함**입니다. 우선순위 분류(P0/P1/P2)와 PR 분할도 의존성 관점에서 합리적입니다.

가장 가치가 큰 Phase는 **Phase 5(이미지 파이프라인)**, **Phase 6(PDF builder)**, **Phase 1(state machine)** 입니다. 이 셋은 **독자 눈에 즉시 보이는 결함을 직접 차단**하기 때문입니다. **Phase 3(claim ledger)**는 가치는 크지만 복잡도와 Researcher latency 비용이 가장 높아 일정 위험이 있습니다.

가장 의문스러운 부분은 **"즉시 시작할 첫 작업"으로 outline.json을 꼽은 것**입니다. 영향도 우선순위로는 정당하지만, 현재 PDF/이미지 결함이 더 즉각적으로 독자를 해칩니다(아래 "전체 평가" 참조). 또한 계획서가 일부 빼먹은 사안이 있습니다(아래 🔍 항목들).

### 누락 1: `insert_images.py`의 확장자 하드코딩
계획서 Phase 5는 `../images/` 경로 문제만 짚었지만, 같은 라인의 `f"{marker_id}.png"` 도 함께 고쳐야 함. diagram provider가 SVG를 출력하면 manifest의 `output_path`(`extract_markers.py:58`이 PNG로 미리 정함)도 같이 업데이트해야 일관됨. **권고**: `extract_markers.py`도 provider/image_type을 알기 전이라 PNG로 가정하고 있는데, `generate_prompts.py`에서 provider 결정 후 manifest의 `output_path` 확장자를 갱신하는 단계를 추가하거나, `insert_images.py`가 manifest의 `output_path`를 그대로 신뢰하도록 변경.

### 누락 2: `generate_bibliography()`가 호출되지 않는 dead code
build_pdf.py:330의 `generate_bibliography` 는 완전히 작성되어 있지만 CLAUDE.md가 `--citations`를 전달하지 않아 호출되지 않는 dead path. 계획서 Phase 6에서 CLAUDE.md 호출 라인만 고쳐도 즉시 살아남. **이 건은 P0 항목 중 가장 적은 노력으로 가장 큰 가시 효과**(참고문헌 페이지 부활).

### 누락 3: extract_markers.py의 marker_id 충돌 가능성
`extract_markers.py:41-49`는 chapter_tag를 `ch01`처럼 정규식으로 추출하지만, 동일 chapter에서 marker description이 동일하면 `re.escape(description)`만으로 같은 marker가 두 번 잡힐 수 있음. `insert_images.py:75`의 `pattern.subn(..., count=1)`이 우연히 보호하지만 manifest와 본문이 1:1 대응한다는 보장은 없음. 큰 문제는 아니지만 schema validation에 "(chapter_file, line_number) 단위 unique" 같은 항목이 들어가면 좋음.

### 누락 4: Architect의 self-validation 자체가 LLM이라 deterministic 아님
계획서는 outline.json을 도입하면서 검증을 코드로 옮긴다는 방향은 맞는데, **Architect의 markdown 출력과 outline.json 출력이 mismatch될 가능성**(LLM이 둘을 동시에 생성)을 완화할 장치가 없음. 권고: outline.json을 **단일 소스**로 두고 markdown은 그로부터 generator script로 변환(또는 그 반대). 두 표현이 분리된 채 LLM이 각각 작성하면 표시되는 챕터 수와 outline.json 챕터 수가 어긋나는 사고가 잠재.

### 누락 5: pipeline_state migration의 "원본 백업" 시점
계획서 §10 롤백 전략에서 "원본을 `.v3.backup.json`으로 저장"한다고 적었지만, **언제 백업을 쓸지** 명시 부족. resume 시 실패 → 자동으로 backup으로 폴백할지, 사용자 명령일지 모호. 권고: migration은 항상 backup → migrate → verify → 실패 시 backup 복원의 atomic 절차로.

### 누락 6: `parse_references.py`의 stdout 거대화 대응이 추상적
Phase 10에서 reference chunking을 추가한다고 했지만, 기존 `parse_references.py`의 호출 인터페이스(stdout JSON)를 어떻게 바꿀지 모호. 권고: chunking을 추가하면서 출력 채널을 stdout JSON → 파일 단위(`output/research/reference_chunks/`)로 분리하고, 호출자(Researcher AGENT.md)에 새 패턴 명시.

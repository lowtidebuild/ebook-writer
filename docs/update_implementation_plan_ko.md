# Ebook Writer Agent 업데이트 구현 계획서

작성일: 2026-04-27
대상 프로젝트: Ebook Writer Agent
목적: 엔지니어링 및 디자인 감사에서 발견된 출력 품질, 토큰 효율, 아키텍처, 기능, 프롬프트 엔지니어링 문제를 실제 구현 가능한 업데이트 작업으로 전환한다.

> 상태 메모: 이 문서는 구현 전 계획 스냅샷이다. 이후 로컬 커밋 `f2598e6`부터 `a12a822`까지의 마일스톤에서 주요 항목이 구현되었으므로, 현재 동작의 기준 문서는 `README.md`, `CLAUDE.md`, `docs/design.md`, `docs/migration_guide.md`를 우선한다.

## 1. 업데이트 목표

이번 업데이트의 핵심 목표는 “프롬프트로 기대하는 파이프라인”을 “코드와 검증기로 보장되는 파이프라인”으로 바꾸는 것이다. 현재 시스템은 연구, 아웃라인, 집필, 편집, 이미지, PDF, 웹 뷰어 단계가 잘 정의되어 있지만, 많은 보증이 `CLAUDE.md`와 각 Agent 프롬프트의 지시에 의존한다. 이 구조는 모델이 지시를 놓치거나 애매하게 해석할 때 최종 산출물 품질이 급격히 흔들릴 수 있다.

업데이트 후에는 다음이 가능해야 한다.

- 아웃라인, 챕터, 인용, 이미지, PDF 결과물을 스키마와 스크립트로 검증한다.
- `pipeline_state.json`이 재개와 부분 재생성의 단일 진실 공급원 역할을 실제로 수행한다.
- Writer Agent가 전체 연구 보고서를 반복해서 읽지 않고 장별 evidence pack만 읽는다.
- 최종 PDF와 웹 뷰어에 실패 placeholder, raw footnote marker, 누락 이미지, 잘못된 챕터 라벨이 남지 않는다.
- 도메인 plugin은 사용자가 명시적으로 선택하거나 확인한 경우에만 활성화된다.
- 비공식 이미지 백엔드나 로컬 프로세스 실행 같은 취약한 경로는 기본값에서 제거하거나 명시 opt-in으로 제한한다.

## 2. 구현 원칙

1. 검증은 프롬프트가 아니라 코드로 강제한다.
2. LLM은 생성과 판단을 맡고, 상태 전이와 artifact 검증은 deterministic script가 맡는다.
3. 대형 컨텍스트를 N개 sub-agent에 반복 주입하지 않는다.
4. 최종 독자에게 보이는 산출물에는 내부 실패 메시지, placeholder, 생성 시스템 표식이 절대 남지 않아야 한다.
5. 기존 pipeline 사용자는 v3 state를 v4 state로 마이그레이션할 수 있어야 한다.
6. 각 단계는 독립적으로 재실행 가능해야 하며, 실패 원인은 구조화된 로그로 남긴다.

## 3. 우선순위 요약

| 우선순위 | 영역 | 작업 | 이유 |
|---|---|---|---|
| P0 | 빠른 가시 결함 | PDF 챕터 라벨, author/citations wiring, viewer 단일 언어 UI, 이미지 경로/확장자 수정 | 독자에게 즉시 보이는 결함이며 다른 구조 변경과 독립적으로 수정 가능 |
| P0 | 상태 관리 | `pipeline_state.json` v4 schema 및 runner 구현 | resume, retry, partial regeneration 안정성의 기반 |
| P0 | 출력 구조 | `outline.json` 및 deterministic outline validator 도입 | 파일명, 의존성, 병렬 scheduling 오류를 차단 |
| P0 | 최종 검증 | 챕터, 인용, 이미지, PDF preflight 검증 강화 | 독자-facing artifact 오류 방지 |
| P1 | 근거 무결성 | `claim_ledger.json`, `chapter_packs` 도입 | 그럴듯하지만 근거 없는 챕터 생성을 줄이되, Researcher latency와 회귀 위험이 커 별도 PR로 도입 |
| P1 | 토큰 효율 | 장별 evidence pack과 reference chunking | 반복 컨텍스트 낭비 축소 |
| P1 | 이미지 | deterministic diagram renderer 기본값 추가 | 비공식 provider 의존도 축소 |
| P1 | 보안 | process sandbox opt-in 전환, reference ignore 확대 | production safety 개선 |
| P2 | 문서 | 프롬프트 중복 제거, README와 design 동기화 | 유지보수성 개선 |

- 단일 언어 viewer HTML에 secondary toggle이 없다.
- manifest `output_path`가 SVG인 이미지도 Markdown에 올바른 경로로 삽입된다.

### Phase 1. 상태 머신을 코드로 이전

목표: `CLAUDE.md`에만 있던 상태 전이 규칙을 실행 가능한 Python runner로 이전한다.

추가 파일:

- `scripts/pipeline_runner.py`
- `scripts/pipeline_state.py`
- `schemas/pipeline_state.v4.schema.json`
- `tests/test_pipeline_state.py`

수정 파일:

- `CLAUDE.md`
- `.claude/commands/generate.md`
- `.claude/commands/resume.md`
- `README.md`
- `README_KO.md`

구현 내용:

1. `pipeline_state.json` v4 schema를 정의한다.

예시:

```json
{
  "schema_version": 4,
  "pipeline": "generate",
  "topic": "string",
  "author": "string",
  "plugin": null,
  "primary_language": "ko",
  "secondary_language": "en",
  "bilingual": false,
  "started_at": "ISO8601",
  "updated_at": "ISO8601",
  "current_step": "research",
  "chapters": [],
  "steps": {
    "research": {
      "status": "pending",
      "retry_count": 0,
      "started_at": null,
      "completed_at": null,
      "artifacts": []
    }
  },
  "gates": {
    "outline": {
      "status": "pending",
      "feedback": null
    },
    "final": {
      "status": "pending",
      "feedback": null
    }
  }
}
```

2. step status는 다음 enum으로 제한한다.

```text
pending
running
completed
skipped
failed
blocked
```

3. `current_step`은 실행 시작 시 갱신하고, step 완료 시 해당 step status와 artifact 목록을 갱신한다.

4. v2/v3 state backfill 로직을 코드로 구현한다.

5. artifact 검증 규칙을 step별로 선언한다.

예시:

```python
REQUIRED_ARTIFACTS = {
    "research": [
        "output/research/research_report.md",
        "output/research/citations.json",
        "output/research/verification_report.json",
    ],
    "outline": [
        "output/outline/table_of_contents.md",
        "output/outline/outline.json",
    ],
}
```

6. `/generate`, `/resume` 프롬프트는 runner 호출과 LLM task dispatch에 집중하도록 축소한다.

수용 기준:

- state 파일 없이 시작하면 v4 state가 생성된다.
- v3 state를 읽으면 누락 필드가 채워진다.
- 완료된 step artifact가 없으면 해당 step 이후가 pending으로 reset된다.
- `author`, `bilingual`이 state에 저장되고 resume 후에도 유지된다.

### Phase 2. Outline을 구조화된 JSON으로 고정

목표: `outline.json`을 단일 소스로 삼고, Markdown outline은 사람이 읽기 위한 파생 산출물로 만든다. 실제 scheduling, 파일명 생성, dependency wave 계산은 모두 `outline.json`만 사용한다.

추가 파일:

- `schemas/outline.schema.json`
- `scripts/validate_outline.py`
- `scripts/render_outline_markdown.py`
- `tests/test_validate_outline.py`

수정 파일:

- `.claude/agents/architect/AGENT.md`
- `CLAUDE.md`

구현 내용:

1. Architect Agent output의 주 산출물을 `output/outline/outline.json`으로 변경한다.

필수 필드:

```json
{
  "book_title": "string",
  "target_audience": "string",
  "chapters": [
    {
      "chapter_id": 1,
      "slug": "intro",
      "title": "string",
      "summary": "string",
      "key_content": ["string"],
      "estimated_words": 3000,
      "dependencies": []
    }
  ]
}
```

2. slug 규칙을 코드로 검증한다.

- lowercase ASCII
- 숫자와 hyphen 허용
- 공백 금지
- 중복 금지

3. dependency graph 검증을 `validate_outline.py`로 이전한다.

검증 항목:

- 모든 dependency가 존재하는 chapter를 참조한다.
- cycle이 없어야 한다.
- chapter ID가 1부터 연속적이어야 한다.
- `estimated_words`가 양수여야 한다.
- `key_content`가 빈 배열이면 실패한다.

4. `render_outline_markdown.py`가 `outline.json`에서 `output/outline/table_of_contents.md`를 생성한다. LLM이 JSON과 Markdown을 따로 작성하게 두지 않는다.

5. Gate 1 승인 후 `state.chapters`는 Markdown 파싱이 아니라 `outline.json`으로 채운다.

6. `validate_outline.py`는 JSON과 Markdown을 모두 입력받는 경우 두 파일의 chapter count, chapter title, dependency가 일치하는지도 확인한다.

수용 기준:

- cycle이 있는 outline은 Step 3 전에 실패한다.
- slug 누락이나 중복은 blocking error로 처리된다.
- Writer output path는 `outline.json`의 slug에서만 생성된다.
- `table_of_contents.md`는 `outline.json`에서 재생성 가능해야 하며, 두 파일이 서로 어긋나면 검증이 실패한다.

### Phase 3. 근거 체계 재설계

목표: 출처 ID가 존재한다는 수준을 넘어, 각 주장이 어떤 근거로 뒷받침되는지 구조화한다.

우선순위: P1. 품질과 토큰 효율에는 중요하지만 Researcher 출력 형식과 Writer 입력 계약을 동시에 바꾸므로 회귀 위험이 크다. Phase 0.5, Phase 1, Phase 2, Phase 4의 안전망을 먼저 만든 뒤 적용한다.

추가 파일:

- `schemas/claim_ledger.schema.json`
- `scripts/build_chapter_packs.py`
- `scripts/validate_claims.py`
- `tests/test_claim_ledger.py`
- `tests/test_chapter_packs.py`

수정 파일:

- `.claude/agents/researcher/AGENT.md`
- `.claude/agents/writer/AGENT.md`
- `.claude/agents/editor/AGENT.md`
- `CLAUDE.md`

구현 내용:

1. Researcher output에 `output/research/claim_ledger.json`을 추가한다.

예시:

```json
{
  "claims": [
    {
      "id": "claim_001",
      "text": "Claude Code can edit local project files after user approval.",
      "topic_tags": ["claude-code", "file-editing", "permissions"],
      "source_ids": [1],
      "supporting_excerpt": "short excerpt or paraphrased evidence",
      "confidence": "verified",
      "allowed_usage": "safe_to_state",
      "notes": ""
    }
  ]
}
```

2. `allowed_usage` enum:

```text
safe_to_state
state_with_hedge
background_only
do_not_use
```

3. `build_chapter_packs.py`가 `outline.json`, `claim_ledger.json`, `citations.json`을 읽고 장별 pack을 생성한다.

출력:

```text
output/research/chapter_packs/ch01_intro.json
output/research/chapter_packs/ch02_setup.json
```

4. Writer Agent input에서 전체 research report, 전체 citations, 전체 verification report 대신 다음을 기본으로 전달한다.

- 해당 장의 `chapter_pack`
- dependency chapter summary
- 필요한 경우 research report path는 fallback으로만 제공

5. Writer는 `safe_to_state`, `state_with_hedge` claim만 본문 주장에 사용할 수 있다.

6. Editor는 본문 footnote와 claim ID를 비교하여 출처가 claim을 실제로 지원하는지 검토한다.

수용 기준:

- Writer가 pack 밖 source ID를 사용하면 validator가 실패한다.
- `do_not_use` claim이 챕터에 등장하면 blocking error가 발생한다.
- claim 근거가 없는 통계, 날짜, API spec, 법률 참조는 blocking issue가 된다.
- fixture 기반 회귀 테스트가 있어야 하며, chapter pack 누락 또는 잘못된 source ID가 전체 책 생성 실패로 이어지기 전에 validation 단계에서 잡힌다.

### Phase 4. 챕터 구조와 인용 검증 강화

목표: PDF 단계까지 넘어가면 안 되는 Markdown 구조 오류를 Step 4에서 확실히 차단한다.

추가 파일:

- `scripts/validate_chapters.py`
- `schemas/chapter_validation.schema.json`
- `tests/test_validate_chapters.py`

수정 파일:

- `.claude/skills/code-example-validator/scripts/validate_references.py`
- `.claude/skills/code-example-validator/SKILL.md`
- `.claude/agents/editor/AGENT.md`
- `.claude/agents/translator/AGENT.md`
- `CLAUDE.md`

구현 내용:

1. `validate_references.py`에 footnote 정의 검증을 추가한다.

검증 항목:

- 본문 marker `[^N]`에 대응하는 footnote definition이 있어야 한다.
- footnote definition이 있는데 본문 marker가 없으면 warning 또는 non-blocking.
- 같은 파일 안에서 같은 footnote definition이 두 번 나오면 blocking.
- citation ID가 `citations.json`에 없어도 blocking.

2. `validate_chapters.py`를 추가한다.

검증 항목:

- H1은 정확히 하나여야 한다.
- H1 형식은 언어별 규칙을 따른다.
  - Korean: `# 제N장: 제목`
  - English: `# Chapter N: Title`
- heading level은 건너뛰지 않는다.
- `[IMAGE: ...]` marker가 Step 5 이후에는 남으면 안 된다.
- `[이미지 생성 실패]`, `[Image generation failed]`는 최종 검증에서 blocking.
- `{N}`, `{번호}`, `{{PLACEHOLDER}}`, `TODO`, `TBD`가 본문에 있으면 blocking.
- raw `[^N]`이 final HTML/PDF 변환 산출물에 남으면 blocking.

3. validation output은 JSON으로 저장한다.

```text
output/logs/validation/chapter_validation.json
output/logs/validation/reference_validation.json
output/logs/validation/code_validation.json
```

4. Editor Agent에는 JSON validation 결과만 전달하고, 긴 stdout 전체를 붙이지 않는다.

5. 번역 단계에도 동일한 구조 검증을 적용한다.

검증 항목:

- primary/secondary chapter count가 일치한다.
- 각 chapter의 H1/H2/H3 개수가 일치한다.
- code block count가 일치한다.
- image reference count가 일치한다.
- footnote marker count와 ID set이 일치한다.
- raw `[IMAGE: ...]`, `[이미지 생성 실패]`, `[Image generation failed]`가 secondary chapter에도 남지 않는다.
- source chapter에 있던 URL과 code fence language tag가 보존된다.

수용 기준:

- orphan footnote marker fixture가 실패한다.
- image failure placeholder fixture가 실패한다.

## 8. 수용 기준

업데이트는 다음 기준을 만족해야 완료로 본다.

1. `./.venv/bin/python3 -m pytest -q`가 통과한다.
2. v3 state 파일을 v4로 마이그레이션할 수 있다.
3. `outline.json` 없이 Step 3가 실행되지 않는다.
4. dependency cycle이 있는 outline은 Step 3 전에 실패한다.
5. `table_of_contents.md`는 `outline.json`에서 생성되며, 두 산출물이 어긋나면 검증이 실패한다.
6. Writer는 장별 chapter pack을 입력으로 받는다.
7. citation ID 존재 여부뿐 아니라 footnote marker/definition 매칭도 검증된다.
8. primary/secondary 번역 챕터의 heading, code block, image reference, footnote marker 구조가 일치한다.
9. `[IMAGE: ...]`, `[이미지 생성 실패]`, `TODO`, `TBD`, raw `[^N]`가 최종 산출물에 남으면 Gate 2에 도달하지 못한다.
10. 한국어 PDF에는 `제N장`, 영어 PDF에는 `Chapter N`이 표시된다.
11. bibliography가 `citations.json`에서 생성된다.
12. author가 state에서 PDF로 전달된다.
13. plugin 미지정 시 legal plugin이 자동 적용되지 않는다.
14. reference materials 전체가 git ignore된다.
15. 단일 언어 viewer에는 secondary language toggle이 보이지 않는다.
16. unsupported Codex image provider는 기본 필수 경로가 아니다.

## 10. 롤백 전략

1. state migration은 항상 `backup → migrate → validate → commit` 순서로 atomic하게 수행한다. 원본은 `output/pipeline_state.v3.backup.json`으로 저장하고, validation 실패 시 즉시 backup을 복원한다.
2. runner 도입 후에도 초기 릴리스에서는 기존 prompt-driven path를 fallback으로 유지한다.
3. image provider routing은 `IMAGE_PROVIDER=codex|gemini|openai|diagram` override로 되돌릴 수 있게 한다.
4. PDF builder 변경은 테스트 fixture로 이전 output과 구조 차이를 비교한다.
5. security policy 변경은 local development override를 제공하되, final Gate에서는 warning을 남긴다.
